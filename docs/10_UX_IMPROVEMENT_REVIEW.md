## 7 套具体方案的取舍

1. 面试页重构：采纳。下一步应把会话、语音、摄像头、计时器、白板同步拆为职责明确的 hooks/runtime 模块；不以文件数量为目标。
2. Setup 三步向导：采纳，属于高收益流程优化。
3. 候场室：采纳，包含麦克风、摄像头、网络、模型 readiness 和本次面试摘要。
4. 报告页增强：部分采纳。只展示真实可解释的数据，不展示没有样本支撑的“超过 65% 用户”。
5. 状态反馈：采纳。状态必须来自真实连接和媒体事件，避免伪状态。
6. 新手引导：部分采纳。不得恢复“API 密钥只保存在浏览器”的旧架构和文案。
7. 白板集成：分阶段采纳。先完成同页切换/分屏和同步状态，再评估画中画与 AI 标注。

## 不直接采纳的内容

- 不在 `localStorage` 或 `sessionStorage` 保存 API Key。设置表单只向本机后端提交新值且绝不回填。
- 不使用无数据来源的转化率、满意度、百分位和提升幅度作为产品事实。
- 不用 `index * 30` 伪造对话时间；需要真实时间戳。
- 不把“已发送”写成“AI 已理解/已评估”，除非服务端有确认事件。
- 不一次性加入雷达图、趋势图、比较、练习模式、暂停、快捷键和 AI 白板标注；这些会扩大状态与测试复杂度。
- 不直接复制报告里的伪组件、图标字符和 CSS，应沿用现有组件、类型和设计系统。

## 建议实施顺序

### P0：先把现有修复闭环

1. 在真实 macOS 环境完整点击一键启动器，验证安装、构建、启动、健康检查、浏览器打开和二次启动复用。自动执行因本地权限审批服务 503 尚未完成。
2. 在 Windows 实机或 CI 验证 `.bat` 的 PowerShell 指纹、端口清理、路径空格和首次安装。
3. 为启动器失败场景补充可读错误和日志入口。
4. 继续拆分 `interview/page.tsx` 的控制器逻辑并补回归测试。

### P1：改善首次成功完成面试的概率

1. 已完成：统一导航。
2. 已完成：Setup 三步向导。
3. 已完成：独立候场室与 readiness 检查。
4. 已完成：结束确认、保存/归档反馈和手动重连。
5. 已完成主要路径：关键错误、权限和备用文字答案文案。

### P2：增强面试过程

1. 倾听/思考/说话与麦克风音量反馈。
2. 底部当前字幕。
3. 白板同页切换或分屏。
4. 面试进度 checkpoint 与恢复。
5. 响应式和无障碍专项验证。

### P3：报告、历史与扩展模式

1. 分数解释、建议优先级和真实时间轴。
2. 已完成：历史搜索、筛选和排序。
3. 评分口径稳定后再做趋势和跨面试比较。
4. 最后评估暂停、快捷键、国际化和暗色主题；不实施练习模式。

## 验收原则

- 每个状态都必须对应真实事件或后端确认。
- 安全改造不能因 UX 便利被回退。
- 新流程必须允许跳过非必要步骤。
- 面试主流程优先保证桌面端稳定，移动端至少不破版且可完成关键操作。
- 新增流程要有失败、取消、重试、刷新和断线场景测试。
- 所有产品指标必须经过埋点或用户测试验证，不能沿用报告中的估算值。
